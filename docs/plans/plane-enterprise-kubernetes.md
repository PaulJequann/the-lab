# Plane Enterprise Kubernetes Deployment Plan

Date: 2026-04-25

This plan captures the agreed deployment shape for Plane Enterprise in this
repository. It is written as an implementation handoff for a fresh agent.

## Goal

Deploy Plane Enterprise to the K3s cluster through the existing GitOps flow:

- Templates are edited under `templates/`.
- `task configure` renders generated files into `kubernetes/`, `ansible/`, and
  `terraform/`.
- ArgoCD auto-discovers `kubernetes/apps/plane` and deploys it into the `plane`
  namespace.

Do not manually edit generated manifests under `kubernetes/` if a matching
template exists.

## Decisions

- Edition: Plane Enterprise, using the official `plane-enterprise` Helm chart.
- Chart repo: `https://helm.plane.so/`.
- Chart version researched on 2026-04-25: `2.3.2`.
- App version researched on 2026-04-25: `2.5.3`.
- Hostname: `plane.local.bysliek.com`.
- Namespace: `plane`.
- Ingress: custom repo-owned `networking.k8s.io/v1` Ingress using Cilium.
- TLS: existing cert-manager `cloudflare-cluster-issuer`, TLS secret
  `plane-tls`.
- Postgres: external database on `services-data`, managed by Ansible.
- Valkey: dedicated chart-managed Valkey for Plane.
- RabbitMQ: dedicated chart-managed RabbitMQ for Plane.
- MinIO: dedicated chart-managed MinIO for Plane uploads.
- OpenSearch: disabled initially.
- Plane AI/PI: disabled initially.
- Silo: enabled.
- Silo connectors: Slack, GitHub, and GitLab disabled initially.
- Email/SMTP: deferred initially.
- Secrets: Infisical path `/kubernetes/plane`.
- Packaging: local Helm wrapper chart with vendored chart dependency and
  `Chart.lock`.

## Chart Gotchas

Plane Enterprise's standard Ingress template only renders when
`ingress.ingressClass == nginx`. This cluster uses Cilium, so do not rely on the
chart's built-in ingress.

Use:

```yaml
plane-enterprise:
  ingress:
    enabled: false
```

Then create a repo-owned Ingress template under the wrapper chart.

Also disable Plane's built-in issuer/certificate generation. The cluster already
has cert-manager and a Cloudflare-backed ClusterIssuer.

Use:

```yaml
plane-enterprise:
  ssl:
    tls_secret_name: plane-tls
    createIssuer: false
    generateCerts: false
```

## Expected Plane Routes

The custom Cilium Ingress should route these paths to the chart-created services:

- `/` -> `plane-app-web:3000`
- `/spaces/` -> `plane-app-space:3000`
- `/god-mode/` -> `plane-app-admin:3000`
- `/api/` -> `plane-app-api:8000`
- `/auth/` -> `plane-app-api:8000`
- `/graphql/` -> `plane-app-api:8000`
- `/marketplace/` -> `plane-app-api:8000`
- `/live/` -> `plane-app-live:3000`
- `/silo/` -> `plane-app-silo:3000`
- `/uploads` or `/uploads/` -> `plane-app-minio:9000`

Use the actual service names generated by the release name. The intended Helm
release name is `plane-app`, so the service names above assume that release
name.

## Implementation Steps

### 1. Add Plane Config

Edit `config.yaml` and add a Plane section near other app config:

```yaml
# plane
plane_namespace: "plane"
plane_domain: "plane.local.bysliek.com"
plane_chart_repo: "https://helm.plane.so/"
plane_chart_name: "plane-enterprise"
plane_chart_version: "2.3.2"
plane_version: "v2.5.3"
plane_release_name: "plane-app"
plane_license_server: "https://prime.plane.so"
plane_storage_class: "nfs-csi"
plane_postgres_db: "plane"
plane_postgres_user: "plane"
plane_postgres_port: 5432
plane_postgres_host: "10.0.10.86"
plane_minio_bucket: "uploads"
plane_minio_storage_size: "3Gi"
plane_valkey_storage_size: "500Mi"
plane_rabbitmq_storage_size: "100Mi"
plane_monitor_storage_size: "100Mi"
plane_silo_enabled: true
plane_silo_slack_enabled: false
plane_silo_github_enabled: false
plane_silo_gitlab_enabled: false
plane_opensearch_enabled: false
plane_pi_enabled: false
plane_email_enabled: false
```

`plane_postgres_host` should point at the `services-data` host. Prefer deriving
it from `terraform_service_lxcs.services-data.ip` in templates when possible.

### 2. Allow ArgoCD Destination

Edit `templates/kubernetes/bootstrap/projects/apps.yaml.j2` and add:

```yaml
    - namespace: '{{ plane_namespace }}'
      server: https://kubernetes.default.svc
```

The existing namespace resource whitelist should already cover the planned
resources: Service, ServiceAccount, ConfigMap, Secret, PVC, Deployment,
StatefulSet, Job, Ingress, Role, RoleBinding, and InfisicalSecret.

### 3. Create Plane Wrapper Chart Templates

Create:

- `templates/kubernetes/apps/plane/Chart.yaml.j2`
- `templates/kubernetes/apps/plane/values.yaml.j2`
- `templates/kubernetes/apps/plane/templates/plane-ingress.yaml.j2`
- `templates/kubernetes/apps/plane/templates/secrets/plane.infisicalsecret.yaml.j2`
- `templates/kubernetes/apps/plane/README.md.j2` or another local note if useful

`Chart.yaml.j2` should define a wrapper chart with dependency:

```yaml
apiVersion: v2
name: plane
description: Plane Enterprise application
type: application
version: 1.0.0
appVersion: "{{ plane_version }}"

dependencies:
  - name: {{ plane_chart_name }}
    version: {{ plane_chart_version }}
    repository: {{ plane_chart_repo }}
```

After rendering, run:

```bash
helm dependency build kubernetes/apps/plane
```

Commit the resulting `kubernetes/apps/plane/Chart.lock` and vendored chart
archive under `kubernetes/apps/plane/charts/`.

### 4. Configure Plane Helm Values

`values.yaml.j2` should nest values under `plane-enterprise:`.

Base values:

```yaml
plane-enterprise:
  planeVersion: {{ plane_version }}

  license:
    licenseServer: {{ plane_license_server }}
    licenseDomain: {{ plane_domain }}

  ingress:
    enabled: false
    minioHost: ""
    rabbitmqHost: ""

  ssl:
    tls_secret_name: plane-tls
    createIssuer: false
    generateCerts: false

  env:
    storageClass: {{ plane_storage_class }}
    pgdb_username: {{ plane_postgres_user }}
    pgdb_name: {{ plane_postgres_db }}
    pgdb_remote_url: ""
    docstore_bucket: {{ plane_minio_bucket }}
    use_storage_proxy: false
    allow_all_attachment_types: false
    enable_drf_spectacular: false

  services:
    postgres:
      local_setup: false
    redis:
      local_setup: true
      volumeSize: {{ plane_valkey_storage_size }}
    rabbitmq:
      local_setup: true
      volumeSize: {{ plane_rabbitmq_storage_size }}
    minio:
      local_setup: true
      volumeSize: {{ plane_minio_storage_size }}
    monitor:
      volumeSize: {{ plane_monitor_storage_size }}
    opensearch:
      local_setup: false
    silo:
      enabled: {{ plane_silo_enabled | tojson }}
      connectors:
        slack:
          enabled: {{ plane_silo_slack_enabled | tojson }}
        github:
          enabled: {{ plane_silo_github_enabled | tojson }}
        gitlab:
          enabled: {{ plane_silo_gitlab_enabled | tojson }}
    pi:
      enabled: {{ plane_pi_enabled | tojson }}

  external_secrets:
    app_env_existingSecret: plane-app-secrets
    live_env_existingSecret: plane-live-secrets
    silo_env_existingSecret: plane-silo-secrets
    doc_store_existingSecret: plane-doc-store-secrets
```

Decide during implementation whether to also externalize
`rabbitmq_existingSecret`. The chart-managed RabbitMQ StatefulSet can consume an
existing secret for credentials, but verify the required keys from the chart
template before wiring it.

### 5. Add InfisicalSecret

Follow the GlitchTip pattern in:

`templates/kubernetes/apps/glitchtip/templates/secrets/glitchtip.infisicalsecret.yaml.j2`

Create an `InfisicalSecret` in namespace `plane`, sync wave `-10`, reading
`/kubernetes/plane`.

It should create the Kubernetes Secrets needed by the chart:

- `plane-app-secrets`
- `plane-live-secrets`
- `plane-silo-secrets`
- `plane-doc-store-secrets`
- optionally `plane-rabbitmq-secrets`, if externalizing RabbitMQ credentials

Required secret material should include generated values for:

- `SECRET_KEY`
- `LIVE_SERVER_SECRET_KEY`
- `AES_SECRET_KEY`
- `SILO_HMAC_SECRET_KEY`
- `MINIO_ROOT_USER`
- `MINIO_ROOT_PASSWORD`
- `AWS_ACCESS_KEY_ID` or chart-required equivalent for doc store
- `AWS_SECRET_ACCESS_KEY` or chart-required equivalent for doc store
- `POSTGRES_PASSWORD`

Do not commit secret values. Do not print secret values into chat or logs.

The `InfisicalSecret` can compose:

- `DATABASE_URL` pointing at `services-data`.
- `REDIS_URL` pointing at the Plane Valkey service.
- `AMQP_URL` pointing at the Plane RabbitMQ service, if RabbitMQ credentials are
  managed by Infisical.

Before implementing, inspect the chart templates for the exact key names
expected by `doc-store.yaml`, `live-env.yaml`, `silo.yaml`, and any RabbitMQ
secret template. Keep the key names aligned with the chart.

### 6. Add Wait-For-Secret Job

Use the existing macro:

`templates/kubernetes/_macros/wait_for_secret.yaml.j2`

Add a wait job for critical secrets so Plane workloads do not start before the
Infisical operator materializes secrets.

At minimum wait for:

- `plane-app-secrets`
- `plane-live-secrets`
- `plane-silo-secrets`
- `plane-doc-store-secrets`

### 7. Add Custom Ingress

Create `templates/kubernetes/apps/plane/templates/plane-ingress.yaml.j2`:

- `apiVersion: networking.k8s.io/v1`
- `kind: Ingress`
- namespace `{{ plane_namespace }}`
- `ingressClassName: cilium`
- annotation `cert-manager.io/cluster-issuer: {{ cert_manager_issuer_name }}`
- host `{{ plane_domain }}`
- TLS secret `plane-tls`
- routes listed in "Expected Plane Routes"

Validate path ordering carefully. More specific prefixes should appear before
`/` in the rendered manifest.

### 8. Provision External Postgres

Extend the existing `services-data` Ansible path to create the Plane database
and role.

Likely files:

- `templates/ansible/playbooks/services-data.yml.j2`
- `templates/ansible/group_vars/services.yaml.j2`
- maybe `templates/ansible/roles/services_data/defaults/main.yaml.j2`

Add a config flag similar to existing phase flags:

```yaml
enable_plane_db: true
```

Fetch the Plane Postgres password from Infisical path `/kubernetes/plane` and
create:

- database: `plane`
- user: `plane`
- password: Infisical `POSTGRES_PASSWORD`
- allowed CIDRs: cluster node network

Do not read or print raw secrets.

### 9. Update Infisical Machine Identity Permissions

The existing identity creation script grants path access to specific Kubernetes
paths. Update it so the relevant identity can read `/kubernetes/plane`.

Likely file:

- `scripts/create-machine-identities.sh`

Look for existing grants such as `/kubernetes/glitchtip/**` and add
`/kubernetes/plane/**`.

### 10. Add Documentation

Keep long-term operations documentation in:

- `docs/plane-operations.md`

That doc should include how to enable:

- Silo connectors later.
- Email/SMTP later.
- Plane AI/PI later.
- OpenSearch later.

The first version of that doc has already been created with this plan.

### 11. Render and Vendor

Run:

```bash
task configure
helm dependency build kubernetes/apps/plane
```

Then verify:

```bash
helm template plane-app kubernetes/apps/plane --namespace plane >/tmp/plane.yaml
yq eval 'select(.kind == "Ingress") | .metadata.name' /tmp/plane.yaml
yq eval 'select(.kind == "StatefulSet") | .metadata.name' /tmp/plane.yaml
yq eval 'select(.kind == "Deployment") | .metadata.name' /tmp/plane.yaml
```

Confirm:

- There is exactly one custom Cilium Ingress.
- No Plane chart Traefik `IngressRoute` is rendered.
- No Plane chart-generated `Issuer` or `Certificate` is rendered.
- No local Postgres StatefulSet is rendered.
- Plane Valkey, RabbitMQ, MinIO, monitor, Silo, and core app workloads render.
- OpenSearch and PI workloads do not render.

### 12. Validation

Run at least:

```bash
yamllint docs/plans/plane-enterprise-kubernetes.md docs/plane-operations.md
task configure
helm dependency build kubernetes/apps/plane
helm template plane-app kubernetes/apps/plane --namespace plane >/tmp/plane.yaml
kubectl apply --dry-run=client -f /tmp/plane.yaml
```

If staging changes for commit, run:

```bash
infisical scan git-changes --staged --redact --no-color --exit-code 1
```

Never paste scan matches or secret values into chat.

## Deployment Order

1. Create required Plane secrets in Infisical under `/kubernetes/plane`.
2. Render repo changes with `task configure`.
3. Run `helm dependency build kubernetes/apps/plane`.
4. Commit and push templates plus generated Kubernetes/Ansible changes.
5. Run the `services-data` Ansible flow to create the Plane Postgres DB/user.
6. Let ArgoCD discover and sync `kubernetes/apps/plane`.
7. Verify:

```bash
kubectl -n plane get pods,svc,ingress,certificate,pvc
kubectl -n argocd get application plane
curl -skI https://plane.local.bysliek.com/
```
